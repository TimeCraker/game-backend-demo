# 📆 2026.03.03.02 下午开发笔记 A

## 主题：实现玩家位置同步逻辑，升级 WebSocket 为 JSON 协议分流处理器

### ✅ 完成工作

1. **协议升级**  
   - 将 WebSocket 通信从纯文本升级为 **JSON 格式**，引入 `GameMessage` 结构体，包含 `type` 字段区分指令（`chat` / `move`）。

2. **消息分流处理**  
   - 在 `HandleWS` 循环中解析 JSON，根据 `type` 调用不同业务逻辑：
     - `chat` → `handleChatLogic`
     - `move` → `handleMoveLogic`

3. **聊天消息持久化与广播**  
   - `handleChatLogic`：将消息存入 `models.Message` 表，并向全服广播 JSON 格式的聊天内容。

4. **玩家位置同步**  
   - `handleMoveLogic`：更新或创建 `models.PlayerPosition` 记录（坐标 X/Y/Z），并向全服广播位置更新 JSON。

5. **广播标准化**  
   - 所有广播数据均为 JSON 格式，方便前端（Unity/网页）直接解析并驱动模型移动或显示聊天。

### 🧪 测试方法
- 启动服务，用 Postman 连接 WebSocket。
- 发送 JSON 格式消息：
  ```json
  {"type":"move","x":10.5,"y":0,"z":20.0}
  {"type":"chat","content":"hello"}
  ```
- 观察数据库 `player_positions` 表是否更新，另一客户端是否收到广播。

### 🧠 技术要点
- **协议设计**：统一的 JSON 结构使后端可扩展更多指令类型（如 `attack`、`join_room`）。
- **数据库同步**：玩家位置实时落盘，支持断线重连后恢复坐标。
- **广播格式**：包含 `type` 字段，前端可针对不同消息渲染不同效果。

# 📆 2026.03.03 晚上开发笔记

## 🎯 主题：心跳机制 + Protobuf 协议升级准备

### ✅ 今日新增功能

- **聊天记录回溯**：新玩家上线时可拉取最近的历史消息（待实现）。

### 🔄 协议升级：从 JSON 到 Protobuf
#### 为什么换？
- **性能**：二进制解析比 JSON 快 10–100 倍，CPU 开销低。
- **体积**：同样数据体积缩小 5–10 倍，节省带宽。
- **严谨**：强类型定义，避免字段错漏。

#### 操作步骤
1. **安装工具**：
   - 下载 `protoc` 编译器，配置环境变量。
   - 安装 Go 插件：`go install google.golang.org/protobuf/cmd/protoc-gen-go@latest`
2. **编写协议文件**：在 `services/auth/proto/game.proto` 定义 `GameMessage` 等消息结构。
3. **生成 Go 代码**：执行 `protoc --go_out=. game.proto`，得到 `game.pb.go`。

- **修改内容**：实现基于 SMTP 协议（TLS/465 端口）的邮件发送函数，并注入 HTML/CSS UI 模板。

### 1. 修复世界状态同步中的幽灵玩家问题

- **问题**：新玩家加入时，服务器会将已离线玩家的历史位置数据（仍存于数据库）同步给新客户端，导致出现“幽灵玩家”。
- **解决方案**：重构 `websocket.go` 中的 `syncWorldState` 逻辑，改为遍历 `GlobalHub` 中的**活跃 WebSocket 连接**，只将当前在线玩家的位置同步给新客户端。

# 📆 2026.03.13 上午开发笔记：项目架构重构与工程规范化

## ✅ 重构内容概览

本次重构以微服务化和工程规范化为目标，对项目布局进行了系统性调整，主要涉及以下五个方面：

1. **统一模块管理**  
   - 将 `go.mod` 从子目录移至**项目根目录**，建立单一 Module 结构，使 `auth`、`gateway` 等服务能互相引用代码，无需发包到云端。

2. **按业务域拆分 handlers**  
   - 将原先庞大的 `handlers` 文件夹按业务域拆分为独立子包：`account`、`send_email` 等，为后续添加商城、背包等功能提供清晰扩展路径。

3. **抽离长连接服务**  
   - 将 WebSocket 和 Hub 相关逻辑迁移至独立的 `gateway` 服务目录，实现连接管理与业务逻辑的物理分离。

4. **共享 Protobuf 定义**  
   - 将 Protobuf 文件及生成的 Go 代码移至项目级共享 `proto` 目录，方便 `auth` 和 `gateway` 服务统一调用，避免重复定义。

5. **消除隐式全局状态**  
   - 彻底移除之前在各 handler 中透传的全局 `DB` 变量，全部替换为明确的 `db.SQLDB` 调用，使数据流向清晰可控，预防并发问题。

6. **修复依赖与路由**  
   - 修复所有因目录调整导致的 import 路径错误，并更新 `main.go` 中的路由注册，确保服务可正常启动。

## 🧠 架构意义解读

- **单一 Module 结构**：项目成为完整的 Monorepo（大仓库），所有微服务（`auth`、`gateway`）均可直接引用内部代码，无需推送到外部包管理仓库，大幅简化开发调试。
- **按业务域分包**：遵循领域驱动设计原则，代码按功能模块自然分离，新增功能只需创建对应包，避免 `handlers` 文件夹无限膨胀。
- **服务边界清晰**：将长连接逻辑抽离为独立 `gateway` 服务，为后续独立部署、水平扩展打下基础。
- **共享协议定义**：统一 `proto` 目录确保 `auth` 与 `gateway` 对消息结构的理解完全一致，消除因定义副本不同步引发的隐蔽 Bug。
- **消除全局状态**：抛弃隐式 `DB` 变量，强制通过显式依赖传递，数据流向透明，为并发安全和单元测试提供保障。

本次重构使项目结构更贴近生产级微服务标准，为后续功能迭代和团队协作奠定了坚实基础。

## 📚 补充说明

### 1. Docker Compose 是什么？
- `docker-compose.yml` 是一个定义多容器应用的工具。你写的这个文件定义了两个服务（`mysql` 和 `redis`），以及它们需要的配置（密码、端口、数据卷等）。
- `docker-compose up -d` 命令会：
  - 如果本地没有指定的镜像，自动从 Docker Hub 拉取。
  - 根据配置创建并启动容器，`-d` 表示后台运行。
  - 创建两个数据卷（`mysql-data` 和 `redis-data`），用于持久化存储数据库文件，即使容器删除数据也不会丢失。

### 2. MySQL 和 Redis 的角色
- **MySQL**：关系型数据库，用于存储结构化数据（如用户信息、游戏记录等）。你设置了 root 密码 `rootpassword` 和初始数据库 `game_dev`，方便后续开发。
- **Redis**：**不是日志记录工具**，而是一个内存中的键值存储系统，常用作**缓存**、**消息队列**或**临时数据存储**（比如存储用户 session、排行榜等）。它基于内存，读写极快，但数据默认不持久化（你挂载了数据卷 `redis-data` 到 `/data`，可以持久化）。

### 3. 脚本文件（.ps1）的作用
- PowerShell 脚本（`.ps1`）可以将常用命令封装起来，例如：
  - `start.ps1`：`docker-compose up -d`
  - `stop.ps1`：`docker-compose down`
  - `logs.ps1`：`docker-compose logs -f`
- 这样你就不用每次手动输入长命令，直接双击或 `.\start.ps1` 即可。

```go
package main

import (
    "log"
    "github.com/gin-gonic/gin"
    "github.com/TimeCraker/game-backend-demo/services/auth/handlers"
    "github.com/TimeCraker/game-backend-demo/services/auth/middleware"
    "github.com/TimeCraker/game-backend-demo/services/auth/models"
    "github.com/redis/go-redis/v9"
    "gorm.io/driver/mysql"
    "gorm.io/gorm"
)

var (
    db  *gorm.DB
    rdb *redis.Client
)

func main() {
    // 连接 MySQL
    dsn := "root:rootpassword@tcp(localhost:3306)/game_dev?charset=utf8mb4&parseTime=True&loc=Local"
    var err error
    db, err = gorm.Open(mysql.Open(dsn), &gorm.Config{})
    if err != nil {
        log.Fatalf("MySQL连接失败: %v", err)
    }
    db.AutoMigrate(&models.User{})
    log.Println("MySQL connected & migrated")

    // 连接 Redis
    rdb = redis.NewClient(&redis.Options{
        Addr: "localhost:6379",
        DB:   0,
    })
    // 可选：Ping 测试
    log.Println("Redis connected")

    // 创建 Gin 引擎
    r := gin.Default()

    // 健康检查
    r.GET("/health", func(c *gin.Context) {
        c.JSON(200, gin.H{"ok": true})
    })

    // 公开路由
    r.POST("/register", handlers.Register(db))
    r.POST("/login", handlers.Login(db))

    // 需要认证的路由组
    auth := r.Group("/api")
    auth.Use(middleware.AuthMiddleware())
    {
        auth.GET("/me", handlers.GetMe(db))
    }

    // 启动服务（默认监听 :8080，这里指定端口 8081）
    if err := r.Run(":8081"); err != nil {
        log.Fatal(err)
    }
}
```

# 📆 2026.03.02 开发笔记：重构数据库层、引入缓存与状态同步雏形

## 🎯 今日目标：让代码更清晰，为高并发打下基础

今天的核心工作是**重构项目结构**，并引入 **Redis 作为缓存层**，为后续的游戏状态同步做准备。我们不仅让代码更加模块化，还初步实现了用户的在线状态管理，并加固了 WebSocket 连接的安全性。

---

## ✅ 具体完成工作

### 1. 模块化重构：将数据库连接抽离到 `db/` 包
- 之前 `main.go` 里直接写了连接 MySQL 和 Redis 的代码，随着服务增多会变得臃肿。
- 现在创建了 `services/auth/db/` 文件夹，将数据库初始化逻辑封装成独立函数。
- **效果**：`main.go` 现在只负责“指挥”——调用 `db.InitMySQL()` 和 `db.InitRedis()`，拿到客户端实例后交给 Gin 路由。代码更清晰，便于单元测试和复用。

```go
// main.go 变得极其简洁
func main() {
    db.InitMySQL()
    db.InitRedis()
    // ... 路由注册
}
```

### 2. 引入 Redis 缓存层（双层存储架构）
- **为什么需要 Redis？**  
  在游戏后端，有些数据是“高频读写”且可以容忍短暂丢失的，比如用户在线状态、实时位置、房间列表。这些数据放在内存数据库（Redis）中，读写速度极快，能支撑高并发。
- **我们做了什么？**  
  在用户登录成功时，除了在 MySQL 中存储账号信息，还会在 Redis 中设置一个 key，例如 `user:online:{userID}`，值为 `1` 并附带过期时间（如 10 分钟）。这样就能快速判断用户是否在线。
- **技术实现**：  
  在 `handlers/login.go` 中，登录验证成功后添加一行：
  ```go
  rdb.Set(ctx, "user:online:"+strconv.Itoa(int(user.ID)), 1, 10*time.Minute)
  ```

### 3. 状态同步雏形：WebSocket 连接时读取在线状态
- 我们在 `services/gateway` 或相关位置实现了 WebSocket 连接处理。
- 当客户端通过 WebSocket 连接时，会携带一个 `token` 参数（从 URL Query 中获取）。
- 服务器解析 token 得到用户 ID，然后去 Redis 查询 `user:online:{userID}`，如果存在且值为 1，则认为用户已在线，允许建立连接；否则拒绝。
- **效果**：初步实现了“在线用户列表”的概念，为后续的实时消息推送打下基础。

### 4. 安全连接：WebSocket 握手时进行 token 验证
- 之前 WebSocket 可能是完全开放的，任何人都能连接。
- 现在我们在 WebSocket 升级前的 HTTP 握手阶段，通过 `c.Query("token")` 获取 token，验证其合法性（解析 JWT），并从 Redis 检查在线状态。
- 只有合法且未过期的 token 才能升级为 WebSocket 连接。
- **好处**：防止未授权用户占用连接资源，确保每个连接都对应一个真实登录的用户。

```go
// 伪代码：WebSocket 路由处理
func wsHandler(c *gin.Context) {
    token := c.Query("token")
    userID, err := utils.ParseToken(token)
    if err != nil {
        c.String(401, "unauthorized")
        return
    }
    // 检查 Redis 在线状态
    val, _ := rdb.Get(ctx, "user:online:"+userID).Result()
    if val != "1" {
        c.String(403, "not online")
        return
    }
    // 升级为 WebSocket 连接
    // ...
}
```

## 🧠 技术要点解析（知其所以然）

### 1. 架构演进：从“面条式”到“模块化”
- **之前**：所有代码挤在 `main.go`，连接逻辑和业务逻辑混在一起，难以维护和扩展。
- **现在**：将数据库连接、模型定义、处理函数、工具函数分层存放，符合**关注点分离**原则。这是走向微服务架构的第一步。

### 2. 双层存储：MySQL + Redis 的经典搭配
- **MySQL**：负责持久化存储，保存用户账号、装备、等级等不可丢失的数据。
- **Redis**：负责高速缓存，保存在线状态、实时位置、临时数据。读写速度可达微秒级，支持高并发。
- 这种架构在游戏、社交、电商等领域广泛应用，能显著降低数据库压力。

### 3. 状态同步雏形
- 通过在 Redis 中标记用户在线状态，我们实现了最简单的状态同步：其他服务（如匹配服务）可以查询某个用户是否在线。
- 后续可以扩展为存储用户当前所在房间、游戏状态等，实现真正的实时同步。

### 4. 安全连接：token 验证的 WebSocket
- WebSocket 是长连接，如果被恶意利用会消耗大量服务器资源。通过在握手时验证 token，我们确保了每个连接都经过身份认证。
- 结合 JWT 的无状态特性，即使服务器集群也能轻松验证。

   ```bash
   git pull origin main --allow-unrelated-histories
   ```
   （如果远程有 README 或 .gitignore，会合并到本地）

4. **添加本地文件并首次推送**
   ```bash
   echo "# My Backend Demo" > README.md
   git add .
   git commit -m "Initial commit"
   git push -u origin main
   ```

## 🔁 四、日常 Git 工作流（重要）

每天开始工作前和结束后，遵循以下步骤：

```bash
# 1. 开始写代码前：同步远程最新版本（避免冲突）
git pull

# 2. 编写/修改代码（在本地进行）

# 3. 完成后：将改动添加到暂存区
git add .

# 4. 提交到本地仓库，并附上清晰说明
git commit -m "本次修改的简要描述"

# 5. 推送到 GitHub 远程仓库
git push
```

> ✅ **注意**：
> - `git add .` 会添加当前目录下所有改动（包括新增、修改、删除）。如果有不需要追踪的文件（如编译产物），务必创建 `.gitignore` 文件并在其中列出。
> - 首次推送时使用 `-u` 参数建立本地分支与远程分支的关联，之后只需 `git push`。
> - 如果在 `git push` 时被拒绝（提示远程有更新），先执行 `git pull` 合并，再重新 `push`。
